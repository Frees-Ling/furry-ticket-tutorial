## 1. HTTP 协议基础

### 1.1 什么是 HTTP？

HTTP（HyperText Transfer Protocol，超文本传输协议）是 Web 的基础协议。它采用**客户端-服务器**模型，客户端发送请求，服务器返回响应。

```text
HTTP 请求-响应模型：

客户端（浏览器/App）                  服务器
      │                                  │
      │────── HTTP 请求 ───────────────→│
      │   (方法 + URL + 头部 + 体)      │
      │                                  │
      │←───── HTTP 响应 ────────────────│
      │   (状态码 + 头部 + 体)          │
      │                                  │
```

### 1.2 HTTP 请求结构

```text
一个 HTTP 请求的完整结构：

请求行：   POST /api/v1/orders HTTP/1.1
请求头：   Host: api.example.com
          Content-Type: application/json
          Authorization: Bearer eyJhbGci...
          User-Agent: Mozilla/5.0
          Accept: application/json

空行：     

请求体：   {"event_id": 1, "quantity": 2}
```

**请求行解析：**

```text
POST        /api/v1/orders         HTTP/1.1
  ↑              ↑                    ↑
 方法           URL                HTTP版本
```

### 1.3 HTTP 响应结构

```text
响应行：   HTTP/1.1 201 Created
响应头：   Content-Type: application/json
          Location: /api/v1/orders/20240501123456
          X-Request-Id: abc-123-def
          Access-Control-Allow-Origin: *

空行：     

响应体：   {"order_no": "20240501123456", "status": "pending_payment"}
```

### 1.4 HTTP/1.1 vs HTTP/2 vs HTTP/3

| 特性 | HTTP/1.1 | HTTP/2 | HTTP/3 |
| ------ | ---------- | -------- | -------- |
| 底层传输 | TCP | TCP | QUIC (UDP) |
| 多路复用 | ❌ 队头阻塞 | ✅ 流复用 | ✅ 流复用 |
| 头部压缩 | ❌ | ✅ HPACK | ✅ QPACK |
| 服务器推送 | ❌ | ✅ | ✅ |
| 连接复用 | ✅ Keep-Alive | ✅ 多路复用 | ✅ 多路复用 |
| 延迟 | 高 | 中 | 低（0-RTT） |
| 浏览器支持 | 全部 | 95%+ | 90%+ |

## 2. HTTP 请求方法

### 2.1 8 种方法详解

| 方法 | 含义 | 幂等 | 安全 | 是否有请求体 |
| ------ | ------ | ------ | ------ | ------------- |
| **GET** | 获取资源 | ✅ | ✅ | ❌ |
| **POST** | 创建资源 | ❌ | ❌ | ✅ |
| **PUT** | 完整替换资源 | ✅ | ❌ | ✅ |
| **PATCH** | 部分修改资源 | ❌ | ❌ | ✅ |
| **DELETE** | 删除资源 | ✅ | ❌ | 通常无 |
| **HEAD** | 获取响应头（无体） | ✅ | ✅ | ❌ |
| **OPTIONS** | 查询支持的方法 | ✅ | ✅ | ❌ |
| **TRACE** | 回显请求（调试） | ✅ | ✅ | ❌ |

**关键概念：**

```text
幂等（Idempotent）：多次执行的结果与一次执行相同
  ✅ PUT /api/users/1 {name: "张三"} 无论执行多少次，结果都是"张三"
  ❌ POST /api/orders 每次执行都会创建一个新订单

安全（Safe）：不会改变服务器状态
  ✅ GET /api/events 只是查询，不修改任何数据
  ❌ DELETE /api/events/1 删除了数据
```

### 2.2 本项目中的方法使用

```python

# GET — 查询资源

# PUT — 完整更新

# PATCH — 部分更新
PATCH  /api/v1/events/{id}         # 更新活动部分字段

# DELETE — 删除资源

## 3. HTTP 状态码

### 3.1 状态码分类

```

1xx — 信息性
  100 Continue    继续发送
  101 Switching Protocols  升级到 WebSocket

2xx — 成功 ✅
  200 OK          请求成功
  201 Created     创建成功（POST 常用）
  204 No Content  成功但无响应体（DELETE 常用）

3xx — 重定向 🔀
  301 Moved Permanently  永久重定向
  302 Found              临时重定向
  304 Not Modified       缓存未过期

4xx — 客户端错误 ❌
  400 Bad Request    请求参数错误
  401 Unauthorized   未认证（未登录）
  403 Forbidden      无权访问（已登录但权限不足）
  404 Not Found      资源不存在
  409 Conflict       冲突（如超卖、重复注册）
  422 Unprocessable Entity  请求体校验失败
  429 Too Many Requests     请求过频繁

5xx — 服务器错误 🔥
  500 Internal Server Error  服务器内部错误
  502 Bad Gateway            网关错误
  503 Service Unavailable    服务暂不可用
  504 Gateway Timeout        网关超时

```text

### 3.2 本项目中使用的状态码

```python
# 本项目中常见状态码场景

# 200 — 请求成功
GET /api/v1/events → 200 + 活动列表

# 201 — 创建成功
POST /api/v1/orders → 201 + 订单信息

# 400 — 参数错误
POST /api/v1/orders {"event_id": -1} → 400 + 错误信息

# 401 — 未认证
GET /api/v1/auth/me (无 Token) → 401 + "未认证"

# 403 — 无权限
DELETE /api/v1/events/1 (普通用户) → 403 + "权限不足"

# 404 — 资源不存在
GET /api/v1/events/999 → 404 + "活动不存在"

# 409 — 冲突
POST /api/v1/orders (库存不足) → 409 + "库存不足"
POST /api/v1/auth/register (用户名存在) → 409 + "用户名已存在"

# 422 — 校验失败
POST /api/v1/auth/register {"email": "invalid"} → 422 + 校验错误详情

# 429 — 请求过频
POST /api/v1/orders (1秒内下单5次) → 429 + "请求过于频繁"
```

### 3.3 RESTful 状态码选择速查表

```text
创建 → 201 Created
获取 → 200 OK（存在） | 404 Not Found（不存在）
列表 → 200 OK（可能为空数组）
更新 → 200 OK（返回更新后数据） | 204 No Content（不返回数据）
删除 → 204 No Content | 200 OK（返回被删除的数据）

参数错误 → 400 Bad Request
未认证 → 401 Unauthorized
无权限 → 403 Forbidden
资源不存在 → 404 Not Found
冲突 → 409 Conflict
请求太频繁 → 429 Too Many Requests
服务器内部错误 → 500 Internal Server Error
```

## 4. HTTP 头部

### 4.1 常用请求头

| 头部 | 示例 | 用途 |
| ------ | ------ | ------ |
| `Host` | `Host: api.example.com` | 指定目标主机（HTTP/1.1 必须） |
| `Content-Type` | `Content-Type: application/json` | 请求体的媒体类型 |
| `Authorization` | `Authorization: Bearer eyJ...` | 认证凭证 |
| `Accept` | `Accept: application/json` | 客户端期望的响应格式 |
| `User-Agent` | `User-Agent: Mozilla/5.0` | 客户端标识 |
| `Origin` | `Origin: http://localhost:5173` | 请求来源（CORS 用） |
| `Referer` | `Referer: http://localhost:5173/checkout` | 来源页面 URL |
| `Cookie` | `Cookie: session=abc123` | 会话 Cookie |
| `X-Request-Id` | `X-Request-Id: uuid` | 请求追踪 ID |

### 4.2 常用响应头

| 头部 | 示例 | 用途 |
| ------ | ------ | ------ |
| `Content-Type` | `Content-Type: application/json` | 响应体的媒体类型 |
| `Content-Length` | `Content-Length: 248` | 响应体长度 |
| `Cache-Control` | `Cache-Control: no-cache` | 缓存策略 |
| `Set-Cookie` | `Set-Cookie: session=abc; HttpOnly` | 设置 Cookie |
| `Access-Control-Allow-Origin` | `Access-Control-Allow-Origin: *` | CORS 允许的来源 |
| `Location` | `Location: /api/orders/123` | 重定向/新建资源位置 |
| `WWW-Authenticate` | `WWW-Authenticate: Bearer` | 认证质询 |
| `X-Request-Id` | `X-Request-Id: uuid` | 请求追踪 ID |

### 4.3 Content-Type 媒体类型

| 媒体类型 | 用途 | 格式 |
| --------- | ------ | ------ |
| `application/json` | JSON 数据 | 结构化数据 |
| `application/x-www-form-urlencoded` | 表单数据 | `key=value&key2=value2` |
| `multipart/form-data` | 文件上传 | 二进制分界 |
| `text/plain` | 纯文本 | 原始字符串 |
| `text/html` | HTML 页面 | 网页 |
| `text/event-stream` | SSE 推送 | 事件流 |
| `application/octet-stream` | 二进制文件 | 字节流 |

## 5. RESTful API 设计

### 5.1 REST 核心原则

REST（Representational State Transfer，表现层状态转移）是一种 API 设计风格，由 Roy Fielding 在 2000 年提出。

```text
REST 六大约束：

1. 客户端-服务器（Client-Server）
   - UI 与数据存储分离
   - 两端可以独立演进

2. 无状态（Stateless）
   - 每个请求包含所有必要信息
   - 服务器不保存客户端状态
   - 示例：每次请求都要带 Token

3. 缓存（Cacheable）
   - 响应隐式或显式标记为可缓存/不可缓存

4. 统一接口（Uniform Interface）
   - 标准化的资源识别和操作
   - 自描述消息
   - HATEOAS（超媒体驱动）

5. 分层系统（Layered System）
   - 中间代理、网关不改变接口

6. 按需代码（Code on Demand，可选）
   - 服务器可向客户端发送可执行代码
```

### 5.2 资源命名规范

```bash
# ✅ 好的 URL 设计

# 使用名词（而非动词）
GET    /api/v1/events              # 活动列表
GET    /api/v1/events/123          # 单个活动

# 使用复数形式
GET    /api/v1/users               # 用户列表
GET    /api/v1/users/456/orders    # 用户 456 的订单

# 层级关系用 / 表示
GET    /api/v1/events/123/tickets  # 活动 123 的票务
GET    /api/v1/orders/789/tickets  # 订单 789 的票务

# 过滤查询用 query parameters
GET    /api/v1/events?category=concert&page=1&size=20
GET    /api/v1/events?start_date=2024-01-01&end_date=2024-12-31

# ❌ 不好的 URL 设计

GET  /api/getEvents                # 动词不应在 URL 中
GET  /api/eventList                # 应使用复数名词
POST /api/createOrder              # 用 POST 方法表示"创建"
GET  /api/events/page/1/size/20    # 分页参数应使用 query
POST /api/deleteOrder/123          # DELETE 方法 + 资源路径
```text

### 5.3 本项目 URL 设计

```

/api/v1/{resource}[/{id}][/{sub-resource}[/{sub-id}]]

前缀：  /api/v1            ← API 版本控制
资源：  /events, /orders    ← 复数名词
ID：    /events/123         ← 路径参数
子资源： /orders/123/tickets ← 层级关系

完整示例：
GET    /api/v1/events                # 活动列表
GET    /api/v1/events/123            # 活动详情
GET    /api/v1/events/123/tickets    # 活动的票务
POST   /api/v1/orders                # 创建订单
GET    /api/v1/orders/456            # 订单详情
GET    /api/v1/users/me              # 当前用户信息
POST   /api/v1/auth/login            # 登录（动词在这里是合理的）

```text

### 5.4 查询参数规范

```bash
# 分页
?page=1&size=20             # 页码+每页条数
?offset=0&limit=20          # 偏移+限制（另一种方案）
?cursor=abc123&limit=20     # 游标分页（大数据量推荐）

# 过滤
?category=concert           # 分类筛选
?status=published           # 状态筛选
?start_date=2024-01-01      # 日期范围
?price_min=100&price_max=500  # 价格范围

# 排序
?sort=price                 # 升序
?sort=-price                # 降序（- 前缀）
?sort=category,price        # 多字段排序

# 字段选择
?fields=id,title,price      # 只返回指定字段

# 搜索
?q=演唱会                    # 关键字搜索
```

## 6. API 版本控制

### 6.1 为什么需要版本控制？

```bash
# 没有版本控制的后果：
# 老板：加个字段 user.phone
# 你：给 UserResponse 加上 phone 字段
# 客户端 A（已上线）崩溃！——它不认识 phone 字段！

# 有版本控制：
# /api/v1/users → 老接口，保持兼容
# /api/v2/users → 新接口，包含 phone 字段
# 客户端 A 继续使用 v1
# 新客户端使用 v2
```text

### 6.2 版本控制策略

| 策略 | 示例 | 优点 | 缺点 |
| ------ | ------ | ------ | ------ |
| URL 路径 | `/api/v1/events` | 简单直观，部署隔离 | URL 更长 |
| 请求头 | `Accept: application/vnd.api+json;version=1` | 版本与 URL 解耦 | 调试不便 |
| 查询参数 | `/api/events?version=1` | 实现简单 | 违反 REST 原则 |
| 子域名 | `v1.api.example.com` | 部署完全隔离 | DNS/证书管理复杂 |

**本项目采用 URL 路径版本控制**（`/api/v1/`），因为它最简单直观，适合中小规模项目。

## 7. CORS（跨域资源共享）

### 7.1 为什么需要 CORS？

浏览器**同源策略**阻止一个源的页面访问另一个源的资源。如果前后端部署在不同域名/端口，就需要 CORS。

```

同源定义：协议 + 域名 + 端口 三者相同

同源示例：
  <http://localhost:5173> 和 <http://localhost:5173/events>  ✅ 同源
  <http://localhost:5173> 和 <http://localhost:8000>         ❌ 端口不同
  <http://localhost:5173> 和 <https://api.example.com>       ❌ 域名/协议不同

```text

### 7.2 简单请求 vs 预检请求

**简单请求**（满足以下所有条件）：

```

- 方法：GET、HEAD、POST
- 头部：Accept、Accept-Language、Content-Language
- Content-Type：application/x-www-form-urlencoded、multipart/form-data、text/plain

```text

**预检请求（Preflight）**：

- 使用 PUT、DELETE、PATCH 等方法
- 发送非简单头部（如 `Authorization`）
- Content-Type 为 `application/json`

```

预检请求流程：

浏览器：
  │
  ├─（OPTIONS 预检请求）─────→ 服务器
  │   Origin: <http://localhost:5173>
  │   Access-Control-Request-Method: POST
  │   Access-Control-Request-Headers: Authorization, Content-Type
  │
  │←──── 预检响应 ────────────│
  │   Access-Control-Allow-Origin: <http://localhost:5173>
  │   Access-Control-Allow-Methods: POST, GET, PUT, DELETE
  │   Access-Control-Allow-Headers: Authorization, Content-Type
  │
  ├─（实际请求 POST /api/orders）──→ 服务器
  │   Authorization: Bearer eyJ...
  │
  │←──（实际响应 201 Created）─────│

```text

### 7.3 本项目的 CORS 配置

```python
# app/main.py — 本项目的 CORS 配置

app.add_middleware(
    CORSMiddleware,
    allow_origins=[                 # 允许的来源
        "http://localhost:5173",    # Vue 开发服务器
        "http://localhost:8000",    # FastAPI 开发服务器
        "https://your-domain.com",  # 生产域名
    ],
    allow_credentials=True,         # 允许携带 Cookie
    allow_methods=["*"],            # 允许所有 HTTP 方法
    allow_headers=["*"],            # 允许所有请求头
)
```
