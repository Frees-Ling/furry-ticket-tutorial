## 0.5.2 请求生命周期

一个完整的下单请求经过以下层级：

### 前端请求流程

```text
用户点击"下单"
  → Vue Router 守卫检查登录状态
  → Pinia store 调用 api.js（Axios 实例）
  → 请求拦截器注入 JWT Token（localStorage）
  → Axios 发起 HTTP POST /api/v1/orders
```

### 后端请求流程

```text
Nginx 接收请求
  → 检查 CORS 预检（OPTIONS）
  → 代理到 FastAPI (127.0.0.1:8000)

FastAPI 中间件栈（按注册顺序）：
  1. CORSMiddleware — 检查 Origin 是否在白名单
  2. TrustedHostMiddleware — 检查 Host 头
  3. LoggingMiddleware — 记录请求方法/路径/耗时
  4. SecurityHeadersMiddleware — 添加 CSP/X-Frame-Options 等
  5. RequestValidationMiddleware — 限制请求体大小

路由层：
  → 路径匹配 /api/v1/orders
  → 依赖注入：get_current_user（解析 JWT）
  → 依赖注入：get_db（获取数据库会话）
  → 依赖注入：get_redis（获取 Redis 客户端）
  → 请求体验证（Pydantic schema）

服务层（OrderService.create_order）：
  1. BehaviorService — 检查频率限制（Redis INCR）
  2. BehaviorService — 检查并发订单数（Redis INCR）
  3. InventoryService — Redis Lua 脚本扣减库存（原子操作）
  4. 数据库操作 — 创建订单记录 + MySQL 乐观锁更新
  5. 发送 WebSocket 通知（座位状态更新）
  6. 返回订单信息
```

### 响应流程

```text
OrderService 返回 OrderResponse
  → FastAPI 自动序列化为 JSON
  → 中间件记录响应状态码和耗时
  → Nginx 返回给前端
  → Axios 响应拦截器处理
  → Pinia store 更新状态
  → Vue 组件响应式更新
```

## 0.5.5 关键设计模式

本系统在多个层次应用了经典设计模式：

### 1. 服务层模式 (Service Layer)

所有业务逻辑封装在 `app/services/` 中，路由层只负责 HTTP 处理：

```python
# ✅ 路由层：只做 HTTP 处理
@router.post("/orders")
async def create_order(
    data: OrderCreate,
    service: OrderService = Depends(get_order_service),
):
    return await service.create_order(data)

# ❌ 错误做法：业务逻辑写在路由中
@router.post("/orders")
async def create_order(data: OrderCreate, db=Depends(get_db)):
    # 校验库存、计算金额、生成订单... → 路由层膨胀
```text

### 2. 单例模式 (Singleton)

`ConnectionManager` 使用模块级全局实例，确保所有 WebSocket 连接共享同一个房间映射：

```python
# app/services/connection_manager.py
manager = ConnectionManager()  # 全局单例（模块导入时创建）
```

类似地，Redis 客户端也是单例（`app/redis_client.py`）。

### 3. 仓库模式 (Repository)，通过依赖注入实现

通过 FastAPI 的 `Depends` 将数据库会话注入到服务中，服务不直接创建会话：

```python
async def get_order_service(
    db: AsyncSession = Depends(get_db),
    redis: Redis = Depends(get_redis),
    current_user: User = Depends(get_current_user),
):
    return OrderService(db, redis, current_user)
```text

### 4. 策略模式 (Strategy)

支付宝 SDK 中，不同支付场景通过参数策略区分：

```python

## 0.5.6 双层防超卖架构

本系统最核心的架构设计是"双层防超卖"：

```text
请求到达
    │
    ▼
┌─────────────────────┐
│ 第一层：Redis Lua   │  ← 毫秒级，纯内存操作
│ 原子扣减库存        │     不涉及数据库
│ 返回 1/0/-1/-2     │
└─────────┬───────────┘
          │ 成功 (1)
          ▼
┌─────────────────────┐
│ 第二层：MySQL 乐观锁│  ← 最终一致性保证
│ version + sold_count│     持久化存储
│ 检查                │
└─────────┬───────────┘
          │ 成功           │ 失败
          ▼                ▼
   ┌──────────┐    ┌──────────────┐
   │ 返回订单  │    │ 归还 Redis   │
   │          │    │ 库存         │
   └──────────┘    └──────────────┘
```

**为什么需要两层？**

- **Redis 层**：处理 99% 的请求，内存操作微秒级响应，扛住高并发
- **MySQL 层**：最终数据一致性保证，防止 Redis 宕机导致数据丢失
- **归还机制**：MySQL 失败时 Redis 自动归还，不丢库存

### 下单数据流

```text
用户 → POST /api/v1/orders
  │
  ├─ 1. BehaviorService.check_order_frequency()
  │     → Redis INCR order_freq:{user_id}
  │     → 频率超过阈值则拒绝
  │
  ├─ 2. BehaviorService.check_concurrent_orders()
  │     → Redis INCR pending_orders:{user_id}
  │     → 并发超过阈值则拒绝
  │
  ├─ 3. InventoryService.deduct_stock()
  │     → Redis EVALSHA 执行 Lua 脚本
  │     → 库存不足返回错误
  │
  ├─ 4. OrderService.create_order() [MySQL]
  │     → INSERT INTO orders (user_id, event_id, quantity, ...)
  │     → UPDATE events SET sold_count+=qty, version+=1
  │       WHERE id=eid AND version=v AND sold_count+qty<=total
  │     → 乐观锁失败则回滚（归还 Redis 库存）
  │
  ├─ 5. 支付流程（异步）
  │     → 返回订单信息 → 前端跳转支付宝
  │     → 支付宝异步回调 → 更新订单状态
  │
  └─ 6. 超时取消（ARQ Worker）
        → 扫描超时未支付订单
        → 取消订单 + 归还库存
```
