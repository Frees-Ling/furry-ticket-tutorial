## 1. Git 核心概念

### 1.1 什么是版本控制？

版本控制系统（Version Control System, VCS）记录文件随时间的变化，让你可以：

- **回溯**：回到任何历史版本
- **分支**：同时开发多个功能
- **协作**：多人并行开发同一项目
- **审计**：知道谁在什么时候改了什么

### 1.2 Git 的特点

```text
Git vs 传统 VCS（如 SVN）：

SVN（集中式）：
        中央服务器（单点故障）
       /     |      \
  开发者1  开发者2  开发者3
  每次操作都需要连接服务器

Git（分布式）：
  每个开发者都有完整的仓库副本！
        远程仓库（GitHub/GitLab）
       /     |      \
  开发者1  开发者2  开发者3
  (完整历史) (完整历史) (完整历史)
  大部分操作在本地完成，无需联网
```

**Git 的核心优势：**

| 特性 | Git | SVN |
| ------ | ----- | ----- |
| 架构 | 分布式 | 集中式 |
| 离线操作 | ✅ 完整功能 | ❌ 大部分不可用 |
| 分支成本 | 极低（指针移动） | 高（目录复制） |
| 合并 | 智能自动合并 | 基础合并 |
| 性能 | 极快（本地操作） | 慢（网络操作） |
| 数据完整性 | SHA-1 校验 | 无 |

### 1.3 Git 的三种状态

Git 文件有三种状态，这是理解 Git 的核心：

```text
                                  ┌─────────────┐
    工作区 (Working Directory) ──→│   git add   │──→ 暂存区 (Staging Area)
        (你看到的文件)           └─────────────┘        (.git/index)
                                                          │
                                                          │ git commit
                                                          ▼
                                                     ┌─────────────┐
                                                     │  本地仓库    │
                                                     │ (.git/objects)│
                                                     └─────────────┘
                                                          │
                                                          │ git push
                                                          ▼
                                                     ┌─────────────┐
                                                     │  远程仓库    │
                                                     │ (GitHub等)   │
                                                     └─────────────┘
```

**三种状态详解：**

| 状态 | 说明 | 所在位置 |
| ------ | ------ | --------- |
| **已修改 (Modified)** | 文件已修改但还未添加到暂存区 | 工作区 |
| **已暂存 (Staged)** | 已标记的修改，准备提交 | 暂存区（.git/index） |
| **已提交 (Committed)** | 已安全保存到本地仓库 | 本地仓库（.git/objects） |

## 2. 安装与配置

### 2.1 安装 Git

```bash
# macOS
brew install git

# Ubuntu/Debian
sudo apt install git

# Windows
# 从 https://git-scm.com 下载安装包
# 安装后使用 Git Bash 终端
```text

### 2.2 首次配置

```bash
# 设置用户名（提交时记录）
git config --global user.name "Your Name"

# 设置邮箱（用于关联 GitHub 账户）
git config --global user.email "your@email.com"

# 设置默认分支名
git config --global init.defaultBranch main

# 设置行尾处理（Windows 推荐）
git config --global core.autocrlf true

# 查看所有配置
git config --list
```

**配置作用域：**

```text
--global    当前用户的所有仓库（最常用）
--system    整台机器的所有用户
--local     当前仓库（默认，在仓库目录内运行）
```

## 3. 基础操作（初始化到提交）

### 3.1 初始化仓库

```bash
# 方式一：在当前目录初始化
cd my-project
git init
# 输出：Initialized empty Git repository in /path/my-project/.git/

# 方式二：克隆已有仓库
git clone https://github.com/user/repo.git
git clone https://github.com/user/repo.git my-folder  # 指定目录名
```text

### 3.2 查看状态

```bash
# 查看工作区和暂存区状态
git status

# 简洁模式
git status -s
# 输出示例：
#  M README.md    ← 已修改（空格+M = 工作区修改）
# M  index.html   ← 已暂存（M+空格 = 暂存区）
# ?? newfile.txt  ← 未跟踪
```

### 3.3 添加与提交

```bash
# 添加文件到暂存区
git add file.txt              # 添加单个文件
git add *.py                  # 添加所有 Python 文件
git add .                     # 添加当前目录所有变化（小心！）
git add -A                    # 添加所有变化（等同 git add --all）

# 提交暂存区到仓库
git commit -m "feat: 添加用户登录功能"

# 跳过暂存区，直接提交已跟踪文件的修改
git commit -a -m "fix: 修复登录bug"

# 修改最后一次提交（未推送时使用）
git commit --amend -m "修正的提交信息"

# 提交后查看历史
git log
git log --oneline             # 简洁格式
git log --graph --oneline     # 图形化显示分支
```text

### 3.4 查看差异

```bash
# 工作区 vs 暂存区（未暂存的修改）
git diff

# 暂存区 vs 最近提交（即将提交的内容）
git diff --staged
git diff --cached            # --staged 的同义词

# 工作区 vs 最近提交
git diff HEAD

# 两个提交之间的差异
git diff commit1..commit2
```

### 3.5 工作流程图

```text
第一次提交：
工作区 → git add → 暂存区 → git commit → 本地仓库
                                          
后续修改：
工作区(修改文件) → git add → 暂存区(新版本) → git commit → 本地仓库(新版本)
                                                
        已跟踪文件可以直接 git commit -a
        新文件必须先 git add
```

## 4. 远程仓库操作

### 4.1 关联远程仓库

```bash
# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 查看远程仓库
git remote -v
# origin  https://github.com/user/repo.git (fetch)
# origin  https://github.com/user/repo.git (push)

# 移除远程仓库
git remote remove origin

# 修改远程仓库 URL
git remote set-url origin https://new-url.com/repo.git
```text

### 4.2 推送与拉取

```bash
# 推送本地分支到远程
git push origin main          # 推送 main 分支
git push -u origin main       # -u 设置上游关联（首次推送使用）
git push                      # 设置上游后可直接推送
git push --all                # 推送所有本地分支
git push --tags               # 推送所有标签

# 从远程拉取（下载并合并）
git pull                      # = git fetch + git merge
git pull origin main          # 拉取指定分支
git pull --rebase             # = git fetch + git rebase（推荐！）

# 只下载不合并
git fetch
git fetch origin
git fetch --all               # 获取所有远程更新
```

### 4.3 克隆与分支

```bash
# 克隆时指定分支
git clone -b dev https://github.com/user/repo.git

# 获取远程分支列表
git branch -r                 # 远程分支
git branch -a                 # 所有分支（本地+远程）

# 基于远程分支创建本地分支
git checkout -b feature-login origin/feature-login
# 或简写
git switch -c feature-login origin/feature-login
```text

## 5. 分支管理

### 5.1 分支的本质

在 Git 中，分支本质上是一个指向某个提交的**指针**。

```

创建分支前：
      ┌─────┐  ┌─────┐  ┌─────┐
      │ C1  │←─│ C2  │←─│ C3  │←─ main (HEAD)
      └─────┘  └─────┘  └─────┘

创建分支：git branch feature
      ┌─────┐  ┌─────┐  ┌─────┐
      │ C1  │←─│ C2  │←─│ C3  │←─ main (HEAD), feature
      └─────┘  └─────┘  └─────┘

切换到新分支：git checkout feature（或 git switch feature）
      ┌─────┐  ┌─────┐  ┌─────┐
      │ C1  │←─│ C2  │←─│ C3  │←─ main, feature (HEAD)
      └─────┘  └─────┘  └─────┘

新分支上提交：git commit -m "add feature"
      ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
      │ C1  │←─│ C2  │←─│ C3  │←─ main
      └─────┘  └─────┘  └─────┘  └─────┘
                       │         │
                       └─────────┘←─ feature (HEAD)

```text

**关键理解：** 创建分支是 O(1) 操作——只是创建一个 40 字节的指针文件，不是复制文件！

### 5.2 分支基本操作

```bash
# 列出分支
git branch                    # 本地分支（* 标记当前分支）
git branch -r                 # 远程分支
git branch -a                 # 所有分支
git branch -v                 # 显示最近提交

# 创建分支
git branch new-feature        # 在当前提交创建
git branch fix-bug abc123     # 在指定提交创建

# 切换分支
git checkout feature          # 传统方式
git switch feature            # 新版方式（推荐）
git switch -c new-branch      # 创建并切换

# 重命名分支
git branch -m old-name new-name

# 删除分支
git branch -d feature         # 删除本地分支（已合并）
git branch -D feature         # 强制删除（未合并也删）
git push origin --delete feature  # 删除远程分支

# 查看分支合并状态
git branch --merged           # 已合并到当前分支的
git branch --no-merged        # 未合并的
```

### 5.3 合并（Merge）

```bash
# 将 feature 分支合并到当前分支
git checkout main             # 先切到目标分支
git merge feature             # 将 feature 合并进来

# 合并时创建合并提交（即使可以快进）
git merge --no-ff feature

# 只合并但不提交（检查冲突后再提交）
git merge --no-commit feature
```text

**合并的类型：**

```

快进合并（Fast-forward）：
  main:  C1 ← C2 ← C3
  feature:           ← C4 ← C5
  合并 main ← feature: main 直接指向 C5

三方合并（3-way Merge）：
  main:  C1 ← C2 ← C3
  feature:  ← C4 ← C5
  结果：创建新提交 M，包含 C3 和 C5 的变更

  示意图：
      C1 ← C2 ← C3 ← M (Merge Commit)
                ↑       ↗
                C4 ← C5

```text

### 5.4 变基（Rebase）

变基是将一个分支的提交"移植"到另一个分支的最新提交之上。

```bash
git checkout feature
git rebase main
```

```text
变基前：
      C1 ← C2 ← C3    (main)
                ↓
                C4 ← C5  (feature)

变基后（feature 重新基于 main）：
      C1 ← C2 ← C3    (main)
                   ↓
                   C4' ← C5'  (feature)
```

**Merge vs Rebase：**

| 特性 | Merge | Rebase |
| ------ | ------- | -------- |
| 历史 | 保留完整分支拓扑 | 线性历史 |
| 可读性 | 保留"从哪里来"的信息 | 更清晰简洁 |
| 冲突处理 | 一次解决所有冲突 | 逐个提交解决 |
| 安全性 | 完全安全 | ❌ 不要对已推送的分支使用 |

**黄金法则：** 永远不要对**已推送**到远程仓库的分支执行 rebase！

### 5.5 Cherry-Pick

将特定提交应用到当前分支：

```bash
# 将某个提交应用到当前分支
git cherry-pick abc123

# 应用多个提交
git cherry-pick abc123 def456

# 不自动提交（解决冲突后手动提交）
git cherry-pick -n abc123
```text

## 6. 解决冲突

### 6.1 冲突产生的原因

当两个分支修改了同一个文件的同一位置时，Git 无法自动知道应该采用哪个版本。

### 6.2 冲突标记

```bash
# 合并冲突后，文件内容如下：
<<<<<<< HEAD
这是 main 分支的版本
=======
这是 feature 分支的版本  
>>>>>>> feature
```

### 6.3 解决冲突步骤

```bash
# 1. 查看哪些文件有冲突
git status

# 2. 手动编辑冲突文件，删除 <<<<<<< ======= >>>>>>> 标记
# 保留需要的代码

# 3. 标记为已解决
git add resolved-file.txt

# 4. 完成合并
git commit

# 或使用可视化工具
git mergetool
```text

### 6.4 中止合并

```bash
# 如果冲突太多，想放弃合并
git merge --abort
git rebase --abort
```

## 7. 撤销与回退

### 7.1 撤销工作区修改

```bash
# 恢复文件到最后一次提交的状态
git checkout -- file.txt
git restore file.txt          # 新版方式（推荐）

# 恢复所有文件
git restore .
```text

### 7.2 撤销暂存区修改

```bash
# 从暂存区移除（文件保留在工作区）
git reset HEAD file.txt
git restore --staged file.txt  # 新版方式（推荐）

# 完全丢弃暂存区修改
git checkout HEAD -- file.txt
```

### 7.3 重置提交（本地操作）

```bash
# --soft：撤销提交，保留工作区和暂存区修改
git reset --soft HEAD~1

# --mixed（默认）：撤销提交和暂存，保留工作区修改
git reset HEAD~1

# --hard：完全丢弃，危险！
git reset --hard HEAD~1

# 重置到指定提交
git reset --hard abc123

# 推送到远程（需要强制推送，小心！）
git push --force-with-lease   # 比 --force 更安全
```text

### 7.4 Revert（安全撤销）

```bash
# 创建一个新的提交来"撤销"指定提交的更改
git revert abc123

# 撤销多个提交
git revert abc123..def456

# 仅添加到暂存区，不自动提交
git revert -n abc123
```

**reset vs revert：**

```text
reset：移动分支指针，"删除"历史

撤销前： C1 ← C2 ← C3 ← HEAD(main)
git reset --hard HEAD~1
撤销后： C1 ← C2 ← HEAD(main)
         C3 被丢弃

revert：创建反向提交，保留历史

撤销前： C1 ← C2 ← C3 ← HEAD(main)
git revert C3
撤销后： C1 ← C2 ← C3 ← C3' ← HEAD(main)
         C3' 是 C3 的"反操作"
```

## 9. Stash（暂存工作）

当你需要切换分支但不想提交当前修改时：

```bash
# 暂存当前修改
git stash
git stash save "WIP: 正在开发登录功能"

# 查看暂存列表
git stash list
# stash@{0}: On feature: WIP: 正在开发登录功能

# 恢复最近暂存
git stash pop                 # 恢复并删除
git stash apply               # 恢复但不删除

# 恢复指定暂存
git stash pop stash@{1}
git stash apply stash@{1}

# 删除暂存
git stash drop stash@{0}      # 删除单个
git stash clear               # 清空所有
```

## 11. Git 内部原理（进阶）

### 11.1 对象模型

Git 是一个**内容寻址文件系统**，其核心是四种对象：

```text
Git 对象类型：

1. Blob（文件内容）
   存储文件数据，无文件名
   SHA-1: blob + 空格 + 内容长度 + \0 + 内容

2. Tree（目录结构）
   存储文件名和对应的 Blob/Tree 引用
   相当于文件系统的目录

3. Commit（提交）
   存储：tree 引用、parent 引用、作者、信息
   每次 git commit 创建一个

4. Tag（标签）
   指向某个 Commit 的固定指针
```

**实战探索：**

```bash
# 查看 .git 目录结构
ls -la .git/
# HEAD          — 当前分支指针
# config        — 仓库配置
# objects/      — 所有数据对象
# refs/         — 分支和标签引用
#   heads/      — 本地分支
#   tags/       — 标签
#   remotes/    — 远程分支

# 查看对象内容
git cat-file -p abc123    # 查看对象内容
git cat-file -t abc123    # 查看对象类型

# 查看提交的完整内容
git cat-file -p HEAD
# tree 8a7b6f5...
# parent 3c2a1b9...
# author Your Name <email> 1710500000 +0800
# committer Your Name <email> 1710500000 +0800
#
#    提交信息
```text

### 11.2 HEAD 引用

```

HEAD 是一个指向当前分支的引用：

正常情况：
HEAD → refs/heads/main → commit abc123

分离 HEAD（detached HEAD）：
HEAD → commit abc123（不指向任何分支）

分离 HEAD 时提交：

- 提交仍然被创建
- 但没有任何分支指向它
- 切换分支后提交可能被 GC 回收

```text

### 11.3 三路合并原理

```

Git 合并时比较三个版本：

O = 原始版本（两个分支分叉前的共同祖先）
A = 当前分支版本（HEAD）
B = 要合并的版本（待合并分支）

合并算法：
对于每个文件：
  for each 修改：
    if A == O and B == O: 不修改（都没改）
    if A != O and B == O: 采用 A 的修改
    if A == O and B != O: 采用 B 的修改
    if A != O and B != O:
      if A == B: 采用相同的修改
      if A != B: 冲突！需要手工解决

```text

| 问题 | 原因 | 解决 |
| ------ | ------ | ------ |
| `fatal: not a git repository` | 当前目录不是 Git 仓库 | 确认在正确的目录或运行 `git init` |
| `fatal: 'origin' does not appear to be a git repository` | 没有配置远程仓库 | `git remote add origin <url>` |
| `Please commit your changes or stash them` | 切换分支前有未提交的修改 | `git stash` 暂存，或先提交 |
| `Merge conflict` | 同一文件同一位置被不同分支修改 | 手动解决冲突后 `git add` |
| `You have divergent branches` | 本地和远程历史分叉 | `git pull --rebase` 变基拉取 |
| `Permission denied (publickey)` | SSH 密钥未配置 | 将公钥添加到 GitHub/GitLab |
| `Updates were rejected because the remote contains work you do not have` | 远程有本地没有的提交 | `git pull` 先拉取再推送 |
| `detached HEAD` | checkout 了某个提交而非分支 | 创建分支：`git switch -c new-branch` |
