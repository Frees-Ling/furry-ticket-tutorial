## 3. 12-Phase Upgrade Roadmap

Each phase built on the previous one, progressively adding capability without breaking existing functionality.

### Phase 1 — TicketType 3-Layer Semantics
Introduced the three-tier ticket type model: **Event > Session > TicketType**. Each ticket type carries its own price, allocation, and inventory rules, enabling complex multi-tier pricing (early bird, VIP, group, student) within a single event.

### Phase 2 — Allocation Module
Implemented the allocation engine supporting three strategies: **lottery** (random draw for oversubscribed events), **reservation** (time-limited holds without payment), and **queue** (first-come-first-served with position tracking). Each strategy shares the same idempotent inventory interface.

### Phase 3 — UserEventProfile + Single-Event Mode
Created the `UserEventProfile` model to track user-level state per event (participation history, allocation status, identity verification). Introduced `single_event` mode restricting users to purchasing tickets for one event at a time (identity events like marathons or conferences).

### Phase 4 — OrderItem Audit Chain + Pricing
Replaced flat order lines with a full **OrderItem** model carrying price snapshots (`unit_price`, `face_value`, `discount_amount`, `final_price`), ticket type references, and allocation strategy metadata. Every order becomes an immutable audit trail of what was sold, at what price, and under which rules.

### Phase 5 — Idempotent Inventory with 3-Identifier Tracing
Implemented inventory operations with three tracing identifiers: `request_id` (client-generated idempotency key), `operation_id` (server-assigned unique operation), and `trace_id` (propagated for distributed debugging). The Redis Lua script and MySQL optimistic lock both enforce exactly-once semantics.

### Phase 6 — Profile/Order Integration
Connected `UserEventProfile` with the order lifecycle. Creating an order updates the profile (last order time, total spent, event participation count). Cancelling/refunding an order decrements counters. This unified view powers the risk engine and user-facing dashboards.

### Phase 7 — Risk Control (4-Domain)
Built the four-domain risk control framework: **frequency** (orders/minute), **velocity** (amount/time window), **concurrency** (pending order cap), and **reputation** (user trust score based on history). Each domain has configurable thresholds and action levels (allow/review/block). Integrated into the order pipeline as a pre-flight check.

### Phase 8 — Frontend 3-Layer UI + Identity Pages
Restructured the Vue 3 frontend into three layers: **Shell** (layout, navigation, global state), **Domain** (event browsing, order management, payment), and **Identity** (user profile, verification documents, event-specific identity submission). Added identity upload and review pages for single-event mode.

### Phase 9 — Operations Console + Config Center
Built an admin operations console with dashboards (revenue, booking rates, system health), user management (search, ban, view history), and a dynamic configuration center enabling operators to tune risk thresholds, allocation parameters, and system settings at runtime without redeployment.

### Phase 10 — Worker Workflow (Event-Driven, Idempotent, Compensation)
Upgraded the ARQ worker from simple cron-like tasks to an event-driven workflow engine with idempotent task execution, automatic retry with exponential backoff, and compensation transactions for failed workflows. Each workflow step writes to an `operation_log` for auditability.

### Phase 11 — Monitoring/Metrics
Integrated Prometheus metrics via `prometheus-fastapi-instrumentator` (request duration, error rates, active connections), added structured logging with `structlog` (correlation IDs, request tracing), and built health check endpoints with dependency status (DB, Redis, Alipay connectivity).

### Phase 12 — Deployment Enhancement + Documentation
Created the dedicated `Dockerfile.worker` for the ARQ worker (separate from the API image), updated production Compose to reference it, hardened the Nginx configuration (increased body size for complex ticket type payloads, ensured WebSocket support), and produced comprehensive architecture documentation covering all 12 phases.

## 4. Key Design Decisions

### 4.1 Dual-Layer Overselling Prevention (Redis + MySQL)

The system never oversells tickets. Two independent layers work together:

```
Request ──▶ Redis Lua DECRBY ──▶ MySQL UPDATE ... WHERE sold + qty <= total
                │                           │
           Fast path                    Slow path
        (atomic stock check)       (optimistic lock with version)
                │                           │
                └─────── Either fail ───────┘
                            │
                      Rollback Redis
                     (WATCH transaction)
```

- **Redis layer** (fast path): An atomic Lua script decrements stock with a user-level quantity limit. Single-digit millisecond latency. Catches 99.9% of oversell attempts.
- **MySQL layer** (slow path): An `UPDATE ... WHERE sold_count + :qty <= total_stock AND version = :ver` optimistic lock catches edge cases (Redis failover, race conditions).
- **Compensation**: If MySQL rejects the order, Redis stock is restored atomically using a WATCH/MULTI/EXEC transaction with an upper bound (never restore above original stock).

**Why not just MySQL?** Redis handles hundreds of thousands of stock checks per second. MySQL row locks would bottleneck at a few thousand. The two-layer approach gives us performance of in-memory with safety of disk-based transactions.

**Why not just Redis?** Redis is volatile — a restart without persistence would lose inventory state. MySQL provides the source of truth; Redis is a cached counter.

### 4.2 3-Layer TicketType Abstraction

```
Event (体育赛事/演唱会)         e.g., "2024 Spring Championship"
  │
  ├── Session (场次)            e.g., "Finals - Afternoon"
  │     │
  │     ├── TicketType          e.g., "VIP", "Regular", "Student"
  │     │     ├── price
  │     │     ├── total_stock
  │     │     ├── max_per_user
  │     │     └── allocation_strategy (lottery|reservation|queue)
  │     │
  │     └── TicketType          e.g., "Early Bird"
  │
  └── Session (场次)            e.g., "Finals - Evening"
```

This three-level hierarchy decouples event management from session scheduling from ticket pricing. An event can have multiple sessions (e.g., different dates or times), and each session can have multiple ticket types with independent pricing, stock, and allocation strategies.

### 4.3 State Machines for All Entities

Every core entity uses an explicit state machine enforced at the database level:

**Order states:**
```
pending_payment ──▶ paid ──▶ refunded
        │
        └──▶ cancelled
```

**Ticket states:**
```
active ──▶ used (verified_at set)
   │
   └──▶ refunded
```

**Payment states:**
```
pending ──▶ completed ──▶ refunded
   │
   └──▶ failed
```

**Allocation states:**
```
pending ──▶ allocated ──▶ claimed
   │          │
   │          └──▶ expired
   └──▶ cancelled
```

State transitions are validated in service layer code and enforced by database constraints. Invalid transitions raise business logic errors.

### 4.4 4-Domain Risk Control

```
                    ┌─────────────────────────────┐
                    │     Risk Evaluation          │
                    │  (scored 0.0 - 1.0)          │
                    ├─────────────────────────────┤
                    │  Thresholds per action level │
                    │  0.0-0.3: allow              │
                    │  0.3-0.7: review (manual)    │
                    │  0.7-1.0: block              │
                    └─────────────────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
┌──────────────┐    ┌──────────────┐    ┌──────────────────┐
│  Frequency   │    │   Velocity   │    │   Concurrency    │
│  orders/min  │    │  amount/sec  │    │  pending orders  │
│  (Redis SS)  │    │  (Redis SS)  │    │  (Redis counter) │
└──────────────┘    └──────────────┘    └──────────────────┘
                                                  │
                                          ┌───────┴───────┐
                                          │   Reputation  │
                                          │  trust score  │
                                          │  (MySQL calc) │
                                          └───────────────┘
```

- **Frequency**: Redis sorted-set sliding window counting requests per user per time unit. Configurable thresholds per endpoint.
- **Velocity**: Total monetary value of orders in a sliding window. Prevents high-value fraud in short time spans.
- **Concurrency**: Number of pending (unpaid) orders per user. Prevents inventory hoarding.
- **Reputation**: Computed score based on historical behavior (completed orders, cancellations, refunds, chargebacks). Updated asynchronously after each order lifecycle event.

Each domain produces a score (0.0-1.0). The aggregate score determines the action: allow, flag for manual review, or block outright.

### 4.5 Event-Driven Worker with Idempotency

The ARQ worker processes background tasks with three guarantees:

1. **At-least-once execution**: Tasks are retried on failure with exponential backoff (1s, 2s, 4s, 8s, ... max 60s).
2. **Idempotent handlers**: Each task checks a deduplication key (`operation_log.task_key`) before performing work. If the key exists, the handler skips execution and returns the previous result.
3. **Compensation transactions**: Failed workflows trigger compensating actions (e.g., if seat release fails after an order expires, a compensation handler retries with fresh state).

```
Order expires ──▶ cancel_expired_orders()
                        │
            ┌───────────┼────────────┐
            ▼           ▼            ▼
    Mark order    Release seats  Send notification
    "cancelled"   (redis+mysql)  (email/SMS)
    (MySQL)           │
                      │
              ┌───────┴───────┐
              ▼               ▼
         Success           Failure
         (done)        ┌──────────────┐
                       │ Compensation │
                       │  retry + heal│
                       └──────────────┘
```

### 4.6 Full Audit Chain with Price Snapshots

Every order line item is an immutable record. The `OrderItem` model stores:

| Field | Purpose |
|-------|---------|
| `ticket_type_id` | Reference to the ticket type (FK) |
| `ticket_type_name` | Snapshot of name at time of order |
| `unit_price` | Price paid per ticket |
| `face_value` | Original face value |
| `discount_amount` | Discount applied |
| `final_price` | Actual amount charged |
| `allocation_strategy` | Strategy used (lottery/reservation/queue) |
| `request_id` | Client idempotency key |
| `operation_id` | Server operation identifier |
| `trace_id` | Distributed tracing ID |
| `created_at` | Timestamp of record creation |

This ensures that even if ticket types or prices change later, every historical order carries its own context for accounting, dispute resolution, and audit.

## 5. Data Flows

### 5.1 Multi-Event Order Creation

```
User selects tickets for Event A (Session 1, VIP) and Event B (Session 2, Regular)

  1. POST /api/v1/orders  { items: [{event_id, ticket_type_id, qty}, ...] }

  2. BehaviorService check:
     - Frequency: last 60s order count < threshold
     - Velocity: total pending order value < threshold
     - Concurrency: pending order count < max
     - Reputation: trust score > minimum

  3. For each order item:
     a. InventoryService.deduct_stock(ticket_type_id, qty)
        - Redis Lua: stock[key] >= qty AND user_limits[key] + qty <= max
        - If fail: rollback ALL prior Redis deductions → return error

  4. OrderService.create_order():
     - BEGIN transaction
     - INSERT INTO orders (user_id, status='pending_payment', total_amount)
     - INSERT INTO order_items (for each item with price snapshots)
     - UPDATE events SET sold_count = sold_count + qty, version = version + 1
       WHERE id = :id AND sold_count + qty <= total_stock AND version = :ver
     - If UPDATE affected 0 rows: ROLLBACK, restore Redis stock → return error
     - COMMIT

  5. UserEventProfile updated (last_order_at, total_spent, order_count)

  6. Return: { order_no, payment_url } (Alipay redirect)

  7. User pays → Alipay async notification → PaymentService.handle_notification()
     - Verify RSA2 signature
     - Check Redis SETNX idempotency (anti-replay)
     - Verify amount matches order
     - UPDATE orders SET status='paid'
     - INSERT tickets (one per ticket type qty)
     - Generate verification codes (TC+YYMMDD+14hex)
     - Send order confirmation via worker
```

### 5.2 Single-Event Identity Application

```
For identity-required events (marathons, conferences):

  1. User visits event detail page
     - Frontend checks UserEventProfile.identity_status

  2. POST /api/v1/events/{id}/identity
     - Upload identity document (ID card, passport, etc.)
     - Submit personal details (name, ID number, etc.)

  3. Admin reviews:
     - GET /api/v1/admin/identity-pending (list pending)
     - POST /api/v1/admin/identity/{id}/approve (or reject)
     - Status transitions: pending → approved / rejected

  4. After identity approved:
     - User can now purchase tickets for this event
     - single_event_mode: purchasing for this event locks out other events
     - Purchase flow proceeds as 5.1

  5. Check-in at venue:
     - POST /api/v1/verification/verify (scan QR code or enter code)
     - Verify ticket exists, is paid, not already used
     - Mark ticket as used (verified_at, verified_by)
```

### 5.3 Lottery Draw Execution

```
For events using lottery allocation:

  1. Lottery open period:
     - Users submit applications (POST /api/v1/events/{id}/lottery/apply)
     - Each application recorded in lottery_pool table
     - Redis tracks application count for real-time display

  2. Draw execution (admin trigger or scheduled task):
     - GET /api/v1/admin/events/{id}/lottery/draw
     - Or worker auto-executes based on event.lottery_draw_at

  3. Algorithm:
     a. Load all eligible applications
     b. Apply weighted random selection:
        - Weight = base_weight + reputation_bonus + loyalty_bonus
        - Early applicants get slight priority (time-weighted)
     c. Select winners up to available ticket count
     d. Reserve tickets for winners (24h to claim)
     e. Notify winners (via worker)

  4. Claim period:
     - Winners claim via POST /api/v1/lottery/{id}/claim
     - Creates pending_payment order (flow 5.1 step 4+)
     - Unclaimed tickets returned to pool after 24h

  5. Waitlist:
     - Non-winners placed in waitlist (ordered by score)
     - As winners expire, next in waitlist is notified
```

### 5.4 Risk Evaluation Pipeline

```
Request ──▶ Pre-filter (IP blacklist, device fingerprint) ──▶ Block if match
                               │
                               ▼
                    Frequency Domain ──▶ Score 1
                    (Redis sorted set, 60s window)
                               │
                               ▼
                    Velocity Domain ──▶ Score 2
                    (Redis sorted set, amount sum)
                               │
                               ▼
                    Concurrency Domain ──▶ Score 3
                    (Redis counter, pending orders)
                               │
                               ▼
                    Reputation Domain ──▶ Score 4
                    (MySQL, historical behavior)
                               │
                               ▼
            ┌──────────────────┴──────────────────┐
            │         Aggregate Score              │
            │   weighted_sum / total_weights       │
            └──────────────────┬──────────────────┘
                               │
                    ┌──────────┼──────────┐
                    ▼          ▼          ▼
                allow(0-0.3) review   block(0.7-1)
                              │
                              ▼
                   Manual review queue
                   (admin operations console)
```

```
                      ┌──────────┐
                      │  Nginx   │  :80 / :443
                      │  1.25    │  SSL termination
                      └────┬─────┘                           External
             ┌──────────────┼──────────────┐                    │
             ▼              ▼              ▼              Network Boundary
        ┌──────────┐ ┌──────────┐ ┌──────────┐            ────┘
        │  FastAPI │ │  Worker  │ │ Frontend │

        └──┬───────┘ └──────────┘ └──────────┘
           │         ▲                         Internal Network
           ▼         │                         ───────────────
     ┌────────┐ ┌────────┐
     │ MySQL  │ │ Redis  │
     │ 8.0    │ │ 7      │
     └────────┘ └────────┘
```

### Container Networking

- **backend** network (internal): mysql, redis, worker, app (no external access)
- **frontend** network (accessible): nginx, app, frontend (nginx is the only entry point)
- MySQL port 3306 and Redis port 6379 are bound to localhost only (for host-based tooling)

### Startup Sequence

1. MySQL and Redis start first (health checks required)
2. FastAPI app starts, runs alembic migrations on startup
3. ARQ worker starts, connects to Redis for task queue
4. Frontend Nginx starts, serves the Vue SPA
5. Main Nginx starts, begins accepting traffic

### Environment Configuration

All configuration is via environment variables (`.env` file):
- `DB_*`: Database connection parameters
- `REDIS_*`: Redis connection and password
- `SECRET_KEY`: JWT signing key
- `CORS_ORIGINS`: Allowed CORS origins
- `ALIPAY_*`: Alipay payment integration
- `DEFAULT_ADMIN_*`: Super admin credentials

### Scaling Considerations

- **Horizontal API scaling**: Add more app containers behind the `api` upstream in nginx config. The upstream `least_conn` directive (in `default.conf`) distributes load to the least busy instance. Redis handles distributed locking.
- **Worker scaling**: Multiple worker containers can run concurrently. Redis ensures task uniqueness via ARQ's built-in deduplication.
- **Database scaling**: MySQL读写分离 (read replicas for reporting/queries). Writes always go to the primary.
- **Caching**: Redis cluster mode for larger deployments. Current single-instance config handles moderate traffic.
